## 1. Introdução

### 1.1 Propósito

Este documento especifica todos os requisitos funcionais do **Sistema de Acompanhamento de Proposições** do Conselho Nacional do Ministério Público (CNMP). O objetivo é fornecer à equipe de desenvolvimento uma descrição completa e inequívoca de todas as funcionalidades que o sistema deve implementar.

**Público-alvo:**
- Equipe de desenvolvimento da fábrica de software
- Analistas de sistemas e arquitetos
- Equipe de testes e qualidade
- Gerentes de projeto
- Stakeholders técnicos do CNMP

### 1.2 Escopo

Este documento cobre os requisitos funcionais de todos os módulos do sistema:
- Autenticação e controle de acesso
- Dashboard executivo com indicadores e gráficos
- Gestão de correições (CRUD completo)
- Gestão de proposições (CRUD completo)
- Workflow de publicação de proposições
- Workflow de comprovação de cumprimento
- Workflow de avaliação de comprovações
- Sistema de busca e filtros avançados
- Sistema de exportação de dados (JSON/PDF)
- Sistema de histórico e auditoria
- Sistema de notificações

**Não estão cobertos neste documento:**
- Requisitos não-funcionais (performance, segurança, etc.) - ver documento específico
- Especificações de interface (wireframes, design) - ver documento específico
- Arquitetura técnica do sistema - ver documento de arquitetura
- Modelo de dados - ver `modelo_de_dados.md`

### 1.3 Convenções do Documento

#### 1.3.1 Identificadores de Requisitos

Cada requisito funcional possui um identificador único no formato:

**RF-[MÓDULO]-[NÚMERO]**

Exemplos:
- `RF-AUTH-001` - Primeiro requisito do módulo de autenticação
- `RF-DASH-005` - Quinto requisito do módulo de dashboard
- `RF-COR-012` - Décimo segundo requisito do módulo de correições

#### 1.3.2 Prioridades

Cada requisito é classificado com uma das seguintes prioridades:

| Prioridade | Sigla | Descrição |
|------------|-------|-----------|
| **Essencial** | [E] | Requisito crítico para o MVP. Sistema não funciona sem ele. |
| **Importante** | [I] | Requisito necessário para operação completa. Pode ser entregue em Fase 2. |
| **Desejável** | [D] | Requisito que agrega valor mas não é essencial. Pode ser entregue em fases posteriores. |

#### 1.3.3 Estrutura de Cada Requisito

Cada requisito funcional é documentado com a seguinte estrutura:

**RF-XXX-NNN: [Nome do Requisito]**
- **Prioridade:** [E/I/D]
- **Descrição:** Descrição detalhada do requisito
- **Entrada:** Dados de entrada necessários
- **Processamento:** Lógica de processamento a ser executada
- **Saída:** Resultado esperado
- **Regras de Negócio:** Referências a regras aplicáveis (RN-XXX)
- **Critérios de Aceite:** Condições que devem ser satisfeitas para considerar o requisito implementado

#### 1.3.4 Referências a Outros Documentos

- **[VIS]** - Documento de Visão do Produto (`visao_do_produto.md`)
- **[MD]** - Modelo de Dados (`modelo_de_dados.md`)
- **[RN]** - Regra de Negócio (definida no modelo de dados ou neste documento)
- **[UC]** - Caso de Uso (definido na seção 3 deste documento)

### 1.4 Visão Geral

O Sistema de Acompanhamento de Proposições implementa um workflow completo de gestão de correições e proposições com três atores principais:

**Atores:**
1. **Administrador (Corregedoria Nacional)** - Gerencia correições, publica proposições, avalia comprovações
2. **Usuário (MP Correicionado)** - Visualiza proposições, envia comprovações
3. **Sistema** - Executa operações automáticas (cálculo de status, notificações, etc.)

**Fluxo Principal:**
```
1. Admin cadastra Correição
2. Admin cadastra e publica Proposições para MP
3. MP recebe notificação e visualiza Proposição
4. MP prepara e envia Comprovação
5. Admin recebe notificação e avalia Comprovação
6. Sistema registra decisão e atualiza status
7. Se parcial/em_andamento: retorna ao passo 2 (republicação)
8. Se finalizada/prejudicada: ciclo encerrado
```

## 2. Requisitos Funcionais

### 2.1 RF-AUTH: Autenticação e Controle de Acesso

#### RF-AUTH-001: Login de Usuário

- **Prioridade:** [E] Essencial
- **Descrição:** O sistema deve permitir que usuários autentiquem-se através de seleção de MP e senha.
- **Entrada:**
  - MP selecionado (dropdown com 27 opções + "Corregedoria Nacional")
  - Senha (campo texto, tipo password)
- **Processamento:**
  1. Validar que MP foi selecionado
  2. Validar que senha foi preenchida
  3. Verificar credenciais no banco de dados
  4. Se válido: criar sessão de usuário
  5. Se inválido: exibir mensagem de erro
- **Saída:**
  - Sucesso: Redirecionamento para dashboard + sessão criada
  - Falha: Mensagem "Credenciais inválidas. Tente novamente."
- **Regras de Negócio:**
  - RN-AUTH-01: Senha deve ter mínimo 8 caracteres
  - RN-AUTH-02: Após 5 tentativas falhas, bloquear conta por 15 minutos
  - RN-AUTH-03: Sessão expira após 8 horas de inatividade
- **Critérios de Aceite:**
  - ✓ Admin consegue fazer login selecionando "Corregedoria Nacional"
  - ✓ Usuário consegue fazer login selecionando seu MP (ex: "MPBA")
  - ✓ Credenciais inválidas exibem mensagem de erro clara
  - ✓ Após login bem-sucedido, usuário é redirecionado para dashboard
  - ✓ Sessão persiste ao navegar entre páginas

---

#### RF-AUTH-002: Logout de Usuário

- **Prioridade:** [E] Essencial
- **Descrição:** O sistema deve permitir que usuários encerrem sua sessão de forma segura.
- **Entrada:** Clique no botão "Sair" no menu de usuário
- **Processamento:**
  1. Destruir sessão atual
  2. Limpar dados de autenticação (localStorage/sessionStorage)
  3. Redirecionar para página de login
- **Saída:** Usuário redirecionado para tela de login sem sessão ativa
- **Regras de Negócio:**
  - RN-AUTH-04: Logout deve limpar todos os tokens de sessão
  - RN-AUTH-05: Após logout, botão "Voltar" do navegador não deve restaurar sessão
- **Critérios de Aceite:**
  - ✓ Botão "Sair" visível em todas as páginas autenticadas
  - ✓ Após logout, tentativa de acessar páginas protegidas redireciona para login
  - ✓ Dados sensíveis são limpos do navegador

---

#### RF-AUTH-003: Controle de Acesso por Perfil

- **Prioridade:** [E] Essencial
- **Descrição:** O sistema deve controlar o acesso às funcionalidades baseado no perfil do usuário (Admin ou Usuário).
- **Entrada:** Perfil do usuário autenticado
- **Processamento:**
  1. Identificar perfil do usuário na sessão
  2. Renderizar menu lateral com itens permitidos para o perfil
  3. Bloquear acesso direto a URLs restritas
- **Saída:** Interface personalizada conforme perfil
- **Regras de Negócio:**
  - RN-AUTH-06: Perfil Admin tem acesso total
  - RN-AUTH-07: Perfil Usuário tem acesso restrito (sem páginas de cadastro, publicação e avaliação)
  - RN-AUTH-08: Tentativa de acesso não autorizado deve redirecionar para página apropriada com mensagem
- **Critérios de Aceite:**
  - ✓ **Admin vê no menu:** Dashboard, Correições, Proposições, Publicar, Avaliar, Cadastro de Correição, Cadastro de Proposição
  - ✓ **Usuário vê no menu:** Dashboard, Correições, Proposições, Enviar Comprovação
  - ✓ Usuário tentando acessar `/publicacao.html` é redirecionado para `index.html` com erro
  - ✓ Usuário tentando acessar `/avaliacao.html` é redirecionado para `index.html` com erro

---

#### RF-AUTH-004: Filtragem de Dados por Perfil

- **Prioridade:** [E] Essencial
- **Descrição:** O sistema deve filtrar dados exibidos conforme o perfil do usuário.
- **Entrada:** Perfil e ramoMP do usuário autenticado
- **Processamento:**
  1. Se perfil = Admin: exibir todos os dados
  2. Se perfil = Usuário: exibir apenas dados do seu ramoMP
- **Saída:** Listas e dropdowns filtrados
- **Regras de Negócio:**
  - RN-AUTH-09: Admin visualiza todas as correições de todos os MPs
  - RN-AUTH-10: Usuário visualiza apenas correições onde `correicao.ramoMP = usuario.ramoMP`
  - RN-AUTH-11: Usuário visualiza apenas proposições vinculadas às suas correições
- **Critérios de Aceite:**
  - ✓ Admin logado vê todas as 27 correições cadastradas
  - ✓ Usuário MPBA logado vê apenas correições onde `ramoMP = 'MPBA'`
  - ✓ Filtros em dashboards e dropdowns respeitam regra de visibilidade
  - ✓ API backend valida permissões (não apenas frontend)

---

#### RF-AUTH-005: Persistência de Sessão

- **Prioridade:** [I] Importante
- **Descrição:** O sistema deve manter a sessão do usuário ativa entre navegações e recarregamentos de página.
- **Entrada:** Token de sessão armazenado localmente
- **Processamento:**
  1. Ao carregar qualquer página, verificar existência de sessão válida
  2. Se sessão válida: restaurar contexto do usuário
  3. Se sessão inválida ou expirada: redirecionar para login
- **Saída:** Usuário permanece autenticado durante navegação normal
- **Regras de Negócio:**
  - RN-AUTH-03: Sessão expira após 8 horas de inatividade
  - RN-AUTH-12: Sessão não deve expirar se usuário estiver ativo
  - RN-AUTH-13: Token deve ser renovado a cada interação
- **Critérios de Aceite:**
  - ✓ Recarregar página (F5) não desloga usuário
  - ✓ Navegar entre páginas mantém sessão ativa
  - ✓ Após 8 horas sem interação, próximo acesso requer novo login
  - ✓ Fechar e reabrir navegador mantém sessão (se dentro do prazo)

#### RF-AUTH-006: Identificação de Usuário na Interface

- **Prioridade:** [I] Importante
- **Descrição:** O sistema deve exibir claramente a identidade do usuário logado na interface.
- **Entrada:** Dados do usuário autenticado (ramoMP, ramoMPNome)
- **Processamento:**
  1. Renderizar área de usuário no cabeçalho
  2. Exibir nome do MP e perfil (Admin/Usuário)
  3. Incluir botão de logout
- **Saída:** Box de usuário visível no topo direito da interface
- **Regras de Negócio:** N/A
- **Critérios de Aceite:**
  - ✓ Nome completo do MP exibido (ex: "MPBA - Ministério Público do Estado da Bahia")
  - ✓ Badge de perfil visível (ex: "Administrador" ou "Usuário")
  - ✓ Botão "Sair" acessível no mesmo componente
  - ✓ Componente visível em todas as páginas autenticadas

### 2.2 RF-DASH: Dashboard Executivo

#### RF-DASH-001: Visualização de Indicadores Principais

- **Prioridade:** [E] Essencial
- **Descrição:** O sistema deve exibir 5 cards de indicadores principais no dashboard com dados em tempo real.
- **Entrada:** Filtro de correição selecionado (ou "Todas")
- **Processamento:**
  1. Buscar todas as correições e proposições aplicando filtro de correição
  2. Aplicar filtro de perfil do usuário (Admin vê tudo, Usuário vê apenas seu MP)
  3. Calcular 5 métricas:
     - **Correições Realizadas:** COUNT(correicoes)
     - **Correições Ativas:** COUNT(correicoes WHERE status='ativo')
     - **Total de Proposições:** COUNT(proposicoes)
     - **Proposições Ativas:** COUNT(proposicoes WHERE statusProcessual != 'encerrada')
     - **Prazo Vencido:** COUNT(proposicoes WHERE statusProcessual='aguardando_comprovacao' AND prazoComprovacao < HOJE)
  4. Renderizar cards com valores e ícones
- **Saída:** 5 cards exibidos em grid responsivo
- **Regras de Negócio:**
  - RN-DASH-01: Cálculos devem refletir estado atual do banco de dados
  - RN-DASH-02: Se filtro de correição ativo, apenas dados dessa correição são considerados
  - RN-DASH-03: Card "Prazo Vencido" deve ter destaque visual (vermelho) se valor > 0
- **Critérios de Aceite:**
  - ✓ Cards exibidos em ordem: Correições Realizadas, Ativas, Total Proposições, Ativas, Prazo Vencido
  - ✓ Ícones apropriados: 🏛️, 🔄, 📄, 🔥, ⚠️
  - ✓ Valores atualizados ao mudar filtro de correição
  - ✓ Card "Prazo Vencido" com fundo vermelho se > 0
  - ✓ Layout responsivo: 5 colunas desktop, 2 colunas tablet, 1 coluna mobile

---

#### RF-DASH-002: Filtro de Dashboard por Correição

- **Prioridade:** [I] Importante
- **Descrição:** O sistema deve permitir filtrar todos os dados do dashboard por uma correição específica.
- **Entrada:**
  - Dropdown de correições (opção padrão: "Todas as Correições")
  - Correições listadas conforme perfil do usuário
- **Processamento:**
  1. Ao carregar dashboard, popular dropdown com correições disponíveis
  2. Ao selecionar correição:
     - Armazenar filtro no estado da aplicação
     - Recalcular todos os indicadores
     - Redesenhar ambos os gráficos
  3. Ao selecionar "Todas as Correições":
     - Limpar filtro
     - Exibir dados agregados
- **Saída:** Dashboard atualizado refletindo o filtro aplicado
- **Regras de Negócio:**
  - RN-DASH-04: Dropdown deve listar apenas correições visíveis ao usuário (conforme perfil)
  - RN-DASH-05: Filtro deve persistir ao navegar entre páginas do dashboard
  - RN-DASH-06: Ao cadastrar nova correição, dropdown deve ser atualizado
- **Critérios de Aceite:**
  - ✓ Dropdown posicionado acima dos cards de indicadores
  - ✓ Primeira opção é "Todas as Correições" (padrão selecionado)
  - ✓ Admin vê todas as correições no dropdown
  - ✓ Usuário vê apenas correições do seu MP
  - ✓ Ao selecionar correição específica, cards e gráficos atualizam dinamicamente
  - ✓ Indicador visual de filtro ativo (ex: badge "Filtro: COR-2024-01")

#### RF-DASH-003: Gráfico de Fluxo de Trabalho

- **Prioridade:** [E] Essencial
- **Descrição:** O sistema deve exibir gráfico de barras mostrando distribuição de proposições por status processual.

- **Processamento:**
  1. Contar proposições por statusProcessual:

     - Aguardando Comprovação
     - Em Análise
     - Encerrada

  3. Renderizar gráfico com 4 barras coloridas verticalmente
- **Saída:** Gráfico de barras verticais com título "Fluxo de Trabalho"
- **Regras de Negócio:**
  - RN-DASH-07: Cores devem seguir padrão:
    - Pendente: amarelo (#ffc107)
    - Aguardando Comprovação: laranja escuro (#ff8c00)
    - Em Análise: azul (#0066cc)
    - Encerrada: azul escuro (#003366)
  - RN-DASH-08: Altura da barra proporcional ao valor (escala automática)
  - RN-DASH-09: Exibir rótulo com número absoluto acima de cada barra
- **Critérios de Aceite:**
  - ✓ Gráfico exibido no lado esquerdo do grid de gráficos
  - ✓ 4 barras verticais com cores distintas
  - ✓ Rótulos claros: "Pendente (4)", "Aguardando Comprovação (5)", etc.
  - ✓ Gráfico atualiza ao mudar filtro de correição
  - ✓ Eixo Y escala automaticamente baseado no valor máximo
  - ✓ Legenda ou tooltip ao passar mouse sobre barra

### 2.3 RF-COR: Gestão de Correições

#### RF-COR-001: Cadastro de Nova Correição

- **Prioridade:** [E] Essencial
- **Descrição:** O sistema deve permitir ao administrador cadastrar uma nova correição com todos os campos obrigatórios e opcionais.
- **Entrada:** Formulário com 12 campos:
  - Número (texto, obrigatório)
  - Ramo do MP (dropdown 27 MPs, obrigatório)
  - Temática (textarea, opcional)
  - Número ELO (texto com máscara, opcional)
  - Tipo (dropdown: Ordinária/Extraordinária/OCD/Inspeção, obrigatório)
  - MP (radio: MPE/MPU, obrigatório)
  - UF (select: single para MPE, multiple para MPU, obrigatório)
  - Data Início (date, obrigatório)
  - Data Fim (date, opcional)
  - Observações (textarea 5000 caracteres, opcional)

#### RF-HIST-001: Estrutura do Array Historico

- **Prioridade:** [E] Essencial
- **Descrição:** Toda proposição deve manter array historico com estrutura padronizada.
- **Entrada:** Criação de proposição
- **Processamento:**
  1. Inicializar `proposicao.historico = []` (array vazio)
  2. Cada entrada no array deve conter:
     ```json
     {
       "tipo": "publicacao" | "comprovacao" | "avaliacao",
       "data": "ISO timestamp",
       "usuario": "string (ramoMP ou 'Corregedoria Nacional')",
       "descricao": "string",
       "observacoes": "string | null",
       "arquivos": ["filename1", "filename2"] | null,  // comprovacao only
       "prazoComprovacao": "YYYY-MM-DD" | null,        // publicacao only
       "statusAnterior": ["processual", "valoracao"],  // publicacao/avaliacao
       "statusNovo": ["processual", "valoracao"]       // publicacao/avaliacao
     }
     ```
  3. Manter ordem cronológica (append-only)
- **Saída:** Estrutura de dados consistente
- **Regras de Negócio:**
  - [MD] RN-HIST-01: Histórico é append-only (nunca deletar/editar)
  - [MD] RN-HIST-02: Ordem cronológica obrigatória
  - [MD] RN-HIST-03: Campos condicionais conforme tipo
- **Critérios de Aceite:**
  - ✓ Proposição nova tem historico = []
  - ✓ Cada entrada tem tipo + data + usuario + descricao
  - ✓ Campos específicos presentes conforme tipo
  - ✓ Timestamps em formato ISO 8601
  - ✓ Array nunca null (vazio se sem histórico)

### UC-001: Realizar Login no Sistema

**Ator(es):** Admin (Corregedoria Nacional), User (MP Correicionado)

**Pré-condições:**
- Sistema disponível e acessível via navegador web
- Usuário possui credenciais válidas

**Fluxo Principal:**
1. Usuário acessa a URL do sistema
2. Sistema exibe página de login com campos "Ramo do MP" e "Senha"
3. Usuário seleciona seu ramoMP no dropdown (27 opções disponíveis)
4. Usuário digita senha no campo correspondente
5. Usuário clica no botão "Entrar"
6. Sistema valida credenciais (protótipo: aceita qualquer senha)
7. Sistema identifica tipo de usuário:
   - Se ramoMP = "Corregedoria Nacional" → tipo = "admin"
   - Caso contrário → tipo = "user"
8. Sistema armazena sessão no sessionStorage
9. Sistema oculta elementos de menu conforme permissões:
   - User: oculta "Avaliar Comprovações", "Nova Correição", "Nova Proposição", "Publicar Proposições"
   - Admin: exibe todos os menus
10. Sistema redireciona para Dashboard
11. Sistema exibe menu do usuário no canto superior direito: "[ramoMP] ▼"

**Fluxos Alternativos:**

**FA-01: Login como Admin**
- 3a. Usuário seleciona "Corregedoria Nacional"
- 7a. Sistema define tipo = "admin" e concede acesso total
- Continua no passo 8

**FA-02: Login como User**
- 3a. Usuário seleciona qualquer MP que não seja "Corregedoria Nacional"
- 7a. Sistema define tipo = "user" e restringe menus
- Continua no passo 8

**Fluxos de Exceção:**

**FE-01: Campos vazios**
- 5a. Usuário clica "Entrar" sem preencher campos
- Sistema exibe validação HTML5: "Preencha este campo"
- Retorna ao passo 3

**Pós-condições:**
- Usuário autenticado no sistema
- Sessão ativa armazenada no sessionStorage
- Dashboard exibido com dados filtrados conforme permissões
- Menu adaptado ao tipo de usuário

**Requisitos Relacionados:** RF-AUTH-001, RF-AUTH-002, RF-AUTH-003, RF-AUTH-006

### UC-010: Ciclo Completo de Proposição (Iterativo)

**Ator(es):** Admin, User

- Sistema operacional com correição e proposição cadastradas
- Proposição com status inicial `['pendente', 'nova']`

**Fluxo Principal: Ciclo Bem-Sucedido (Finalizada na Primeira Tentativa)**

1. **Fase 1: Publicação (Admin)**
   - Admin acessa "Publicar Proposições"
   - Seleciona correição
   - Marca proposição(ões) pendente(s)
   - Define prazoComprovacao (ex: 30 dias)
   - Adiciona observações (opcional)
   - Confirma publicação
   - Sistema atualiza status: ['aguardando_comprovacao', 'nova']
   - Sistema registra entrada de 'publicacao' no historico
   - Sistema define prazoComprovacao e dataPublicacao

2. **Fase 2: Comprovação (User)**
   - User acessa "Enviar Comprovação"
   - Identifica proposição publicada (com prazo definido)
   - Clica "Adicionar"
   - Preenche descrição do adimplemento (até 7.500 chars)
   - Adiciona observações (opcional)
   - Anexa arquivos comprobatórios (PDFs, imagens, planilhas)
   - Envia comprovação
   - Sistema atualiza status: ['em_analise', 'nova']

   - Admin acessa "Avaliar Comprovações"
   - Identifica proposição em análise
   - Clica "Avaliar"
   - Analisa descrição, arquivos anexados e histórico
   - Seleciona decisão: ✅ Finalizada
   - Redige parecer positivo
   - Submete avaliação
   - Sistema atualiza status: ['encerrada', 'finalizada']
   - Sistema registra entrada de 'avaliacao' no historico
   - **Ciclo finalizado com sucesso ✓**

4. **Pós-Ciclo:**
   - Proposição removida de todas as filas de trabalho
   - Correição recalcula status (pode tornar-se 'inativo' se todas concluídas)
   - Dashboard atualizado
   - Histórico completo preservado: pub1 → comp1 → aval1

**Fluxos Alternativos:**

**FA-01: Ciclo com Avaliação Parcial (Republicação)**

1. **Fases 1-2:** Idênticas ao fluxo principal
2. **Fase 3: Avaliação Parcial (Admin)**
   - Admin seleciona decisão: ⚠️ Parcial
   - Parecer indica cumprimento parcial e explica pendências
   - Sistema atualiza status: ['pendente', 'parcial']
   - **IMPORTANTE:** Status processual volta para 'pendente' (não 'parcial')
   - Histórico preserva valoração 'parcial' na entrada de avaliacao
   - Proposição reaparece em "Publicar Proposições"

3. **Fase 4: Republicação (Admin)**
   - Admin acessa "Publicar Proposições"
   - Identifica proposição com valoração 'parcial'
   - Marca proposição
   - Define NOVO prazoComprovacao (pode ser diferente do anterior)
   - Adiciona novas observações (opcional)
   - Confirma republicação
   - Sistema atualiza status: ['aguardando_comprovacao', 'parcial']
   - Sistema registra NOVA entrada de 'publicacao' no historico (não substitui anterior)
   - **Histórico agora:** pub1 → comp1 → aval1(parcial) → pub2

4. **Fase 5: Nova Comprovação (User)**
   - User acessa "Enviar Comprovação"
   - Vê proposição republicada
   - Preenche descrição complementar (focando nas pendências apontadas)
   - Anexa novos arquivos
   - Envia nova comprovação
   - Sistema atualiza status: ['em_analise', 'parcial']
   - Sistema registra NOVA entrada de 'comprovacao' no historico
   - **Histórico agora:** pub1 → comp1 → aval1(parcial) → pub2 → comp2

5. **Fase 6: Reavaliação (Admin)**
   - Admin avalia nova comprovação
   - Seleciona decisão: ✅ Finalizada (desta vez completo)
   - Sistema atualiza status: ['encerrada', 'finalizada']
   - Sistema registra NOVA entrada de 'avaliacao' no historico
   - **Ciclo finalizado após 2 iterações ✓**
   - **Histórico completo:** pub1 → comp1 → aval1(parcial) → pub2 → comp2 → aval2(finalizada)

**FA-02: Ciclo com Avaliação Em Andamento (Republicação)**

Fluxo idêntico ao FA-01, mas:
- Fase 3: Admin seleciona ❌ Em Andamento
- Status: ['pendente', 'em_andamento']
- Parecer explica não cumprimento
- Republicação obrigatória com novo prazo
- Iterações continuam até adimplência ou até Admin decidir prejudicada

**FA-03: Ciclo Encerrado como Prejudicada**

1. **Fases 1-2:** Idênticas ao fluxo principal
2. **Fase 3: Avaliação Prejudicada (Admin)**
   - Admin seleciona decisão: 🚫 Prejudicada
   - Parecer explica motivo:
     - Proposição superada por mudança legal
     - Não mais aplicável ao contexto
     - Correição encerrada prematuramente
     - Outro motivo fundamentado
   - Sistema atualiza status: ['encerrada', 'prejudicada']
   - **Ciclo finalizado sem comprovação de cumprimento ✓**

**FA-04: Comprovação com Rascunho Intermediário**

1. **Fase 2a: Salvar Rascunho (User)**
   - User acessa "Enviar Comprovação"
   - Começa a preencher formulário
   - Percebe que precisa buscar mais documentos
   - Clica "💾 Salvar como Rascunho"
   - Sistema salva em `proposicao.rascunhos[]`
   - Status permanece: ['aguardando_comprovacao', valoracao]
   - Proposição continua na lista com indicador "📝 Rascunho"

2. **Fase 2b: Retomar e Enviar (User - posterior)**
   - User retorna à "Enviar Comprovação"
   - Clica "Editar" (em vez de "Adicionar")
   - Sistema auto-carrega rascunho
   - User completa com informações faltantes
   - Clica "📤 Enviar Comprovação"
   - Sistema limpa rascunho
   - Status muda para: ['em_analise', valoracao]
   - Continua com Fase 3 (Avaliação)

**FA-05: Prazo Vencido (User Envia Atrasado)**

1. **Fase 1:** Publicação com prazo definido
2. **Intervalo:** User NÃO envia comprovação dentro do prazo
3. **Sistema:** Detecta `prazoComprovacao < hoje`
   - Dashboard incrementa card "Prazo Vencido"
   - Tabelas destacam linha com background vermelho + ⚠️
   - Alerta na página de comprovação: "⚠️ PRAZO VENCIDO - Vencido há X dia(s)"
4. **Fase 2 (Atrasada):** User envia comprovação mesmo com prazo vencido
   - Sistema ACEITA comprovação (não bloqueia)
   - Status muda para: ['em_analise', valoracao]
   - Histórico registra data real de envio (após prazo)
5. **Fase 3:** Admin avalia considerando atraso
   - Parecer pode mencionar descumprimento de prazo
   - Decisão pode ser influenciada pelo atraso

**Fluxos de Exceção:**

**FE-01: User Não Envia Comprovação (Indefinidamente)**
- Após Fase 1 (publicação), user nunca envia comprovação
- Status permanece: ['aguardando_comprovacao', valoracao]
- Proposição acumula atraso (prazo vencido)
- Dashboard e tabelas sinalizam problema
- Admin pode:
  - Republicar com novo prazo (tentar novamente)
  - Avaliar como em_andamento sem aguardar comprovação (futuro)
  - Marcar como prejudicada se deixar de ser aplicável

**FE-02: Admin Não Avalia Comprovação (Indefinidamente)**
- Após Fase 2 (comprovação enviada), admin não avalia
- Status permanece: ['em_analise', valoracao]
- Proposição fica "presa" na fila de avaliação
- User não pode submeter nova comprovação
- Correição permanece 'ativa'
- Solução: Admin deve processar fila de avaliações

**FE-03: Múltiplas Iterações (Em Andamento Repetido)**
- User envia comprovação inadequada múltiplas vezes
- Admin avalia como em_andamento repetidamente
- Ciclo: pub1 → comp1 → aval1(inad) → pub2 → comp2 → aval2(inad) → pub3 → ...
- Histórico cresce com múltiplas entradas
- Admin pode eventualmente marcar como prejudicada para encerrar

**Pós-condições:**
- **Ciclo Finalizada:** Proposição encerrada com sucesso, correição pode tornar-se inativa
- **Ciclo Parcial/Em Andamento:** Proposição volta para pendente, aguarda republicação
- **Ciclo Prejudicada:** Proposição encerrada sem cumprimento
- **Histórico:** Completo e auditável, preservando TODAS as interações
- **Timeline:** Visual mostra evolução completa com cores e ícones
- **Dashboard:** Sempre atualizado refletindo estado real
- **Rastreabilidade:** 100% das transições documentadas

**Requisitos Relacionados:** RF-PUB-*, RF-COMP-*, RF-AVAL-*, RF-HIST-*, RF-NOTIF-*

## 4. Regras de Negócio Consolidadas

Esta seção consolida TODAS as regras de negócio (RN-*) referenciadas nos 105 requisitos funcionais documentados nas seções anteriores. As regras estão organizadas por módulo funcional para facilitar referência e manutenção.

### 4.1. Módulo de Autenticação (RN-AUTH)

**RN-AUTH-01: Perfis de Usuário**
- **Descrição:** O sistema possui exatamente 2 perfis: Admin (Corregedoria Nacional) e User (Órgão Correicionado)
- **Justificativa:** Separação clara de responsabilidades entre órgão fiscalizador e fiscalizado
- **Impacto:** Controla visibilidade de menus, páginas e funcionalidades

**RN-AUTH-02: Autenticação Simplificada (Protótipo)**
- **Descrição:** Sistema aceita qualquer credencial válida sem validação real de senha
- **Justificativa:** Protótipo funcional sem backend - foco em workflow e UI
- **Impacto:** NÃO deve ser usado em produção - requer integração com sistema de autenticação corporativo

**RN-AUTH-03: Persistência de Sessão**
- **Descrição:** Sessão do usuário é mantida em localStorage para persistência entre navegações
- **Justificativa:** Evita re-login ao navegar entre páginas standalone (index.html, proposicoes.html, etc.)
- **Impacto:** Auto-login funciona em todas as páginas do sistema

**RN-AUTH-04: Controle de Acesso por Perfil**
- **Descrição:** Admin vê TODAS as funcionalidades; User vê apenas funcionalidades não-administrativas
- **Justificativa:** Segregação de funções conforme hierarquia institucional
- **Impacto:** Páginas administrativas redirecionam usuários não-autorizados para index.html

### 4.2. Módulo de Dashboard (RN-DASH)

**RN-DASH-01: Filtragem de Dados por Correição**
- **Descrição:** Dashboard pode exibir estatísticas de todas as correições (padrão) ou de uma correição específica
- **Justificativa:** Permite foco em revisão específica durante operação ativa
- **Impacto:** Todos os cards e gráficos recalculam dinamicamente ao alterar filtro

**RN-DASH-02: Visibilidade Baseada em Permissão**
- **Descrição:** Admin vê todas as correições; User vê apenas correições do seu ramoMP
- **Justificativa:** Correicionado não deve visualizar dados de outros MPs
- **Impacto:** Filtro de correição é pré-filtrado conforme perfil do usuário

**RN-DASH-03: Contadores em Tempo Real**
- **Descrição:** Todos os 5 cards (Correições Realizadas, Ativas, Total Proposições, Ativas, Prazo Vencido) atualizam após qualquer operação
- **Justificativa:** Dashboard deve refletir estado atual do sistema
- **Impacto:** Funções de publicação, comprovação e avaliação DEVEM chamar updateDashboard()

**RN-DASH-04: Gráficos Bidimensionais**
- **Descrição:** Sistema exibe 2 gráficos separados: Fluxo de Trabalho (4 status processuais) e Valoração (5 status de avaliação)
- **Justificativa:** Status bidimensional [processual, valoracao] requer visualização separada
- **Impacto:** Alterações de status devem atualizar ambos os gráficos

**RN-DASH-05: Escalabilidade Visual dos Gráficos**
- **Descrição:** Barras dos gráficos auto-escalam altura com base no maior valor, largura distribui igualmente entre categorias
- **Justificativa:** Gráficos permanecem legíveis independente da distribuição de dados
- **Impacto:** Algoritmo de renderização deve recalcular proporções dinamicamente

### 4.3. Módulo de Correições (RN-COR)

**RN-COR-01: Status Calculado Automaticamente**
- **Descrição:** Status de correição ('ativo' ou 'inativo') é calculado com base nas proposições vinculadas
- **Justificativa:** Evita inconsistência - status manual poderia divergir da realidade
- **Impacto:** Status NÃO é campo editável; recalculado por calcularStatusCorreicao()

**RN-COR-02: Critério de Correição Ativa**
- **Descrição:** Correição é 'ativa' se possui ao menos uma proposição com valoração diferente de 'finalizada' ou 'prejudicada'
- **Justificativa:** Correição só está completa quando todas as proposições estão resolvidas
- **Impacto:** Encerramento de proposições como finalizada/prejudicada pode tornar correição inativa

**RN-COR-03: Seleção de UF por Tipo de MP**
- **Descrição:** MPE permite uma única UF; MPU permite múltiplas UFs
- **Justificativa:** MPE é estadual (uma jurisdição); MPU é federal (múltiplas jurisdições)
- **Impacto:** Formulário ajusta dinamicamente (toggleUFMultiple) ao alterar campo MP

**RN-COR-04: Formato de Número ELO**
- **Descrição:** Número ELO deve seguir formato NNNNNNN-DD.AAAA.J.TT.OOOO
- **Justificativa:** Padronização de identificação no sistema ELO nacional
- **Impacto:** Validação pode ser implementada no frontend ou backend

**RN-COR-05: Tipos de Correição Padronizados**
- **Descrição:** 4 tipos válidos: 'Ordinária', 'Extraordinária', 'OCD', 'Inspeção'
- **Justificativa:** Tipologia oficial do CNMP
- **Impacto:** Dropdown deve conter exatamente essas 4 opções

**RN-COR-06: Ordenação da Tabela**
- **Descrição:** Tabela de correições suporta ordenação por 5 colunas: Número, Total Proposições, Pendente, Em Análise, Prazo Vencido
- **Justificativa:** Facilita priorização de trabalho (ex: ordenar por Prazo Vencido DESC)
- **Impacto:** Clique no header alterna ASC/DESC com indicador visual

**RN-COR-07: Filtro de Status**
- **Descrição:** Usuário pode filtrar tabela para ver: Todas, Apenas Ativas, ou Apenas Inativas
- **Justificativa:** Operação diária foca em ativas; consulta histórica foca em inativas
- **Impacto:** Filtro trabalha em conjunto com busca textual (lógica AND)

### 4.4. Módulo de Proposições (RN-PROP)

**RN-PROP-01: Vínculo Obrigatório com Correição**
- **Descrição:** Toda proposição DEVE ter correicaoId válido
- **Justificativa:** Proposições não existem isoladamente - derivam de correições
- **Impacto:** Formulário de cadastro exige seleção de correição; deleção de correição deve considerar proposições vinculadas

**RN-PROP-02: Status Bidimensional**
- **Descrição:** Status é array [statusProcessual, valoracao] com 2 dimensões independentes
- **Justificativa:** Workflow (onde está) é diferente de qualidade (como foi avaliado)
- **Impacto:** Todas as funções de consulta devem usar índices [0] e [1] corretamente

**RN-PROP-03: Tipos de Proposição**
- **Descrição:** 2 tipos válidos: 'Determinação' (obrigatória) e 'Recomendação' (orientativa)
- **Justificativa:** Distinção jurídica entre ordem vinculante e orientação não vinculante
- **Impacto:** Tipo influencia como correicionado trata a proposição

**RN-PROP-04: Prazo de Comprovação**
- **Descrição:** prazoComprovacao só é definido quando proposição é publicada (entra em aguardando_comprovacao)
- **Justificativa:** Prazo é responsabilidade da Corregedoria Nacional ao publicar
- **Impacto:** Campo é null até publicação; proposições pendentes não têm prazo

**RN-PROP-05: Histórico Append-Only**
- **Descrição:** Array historico[] só recebe novos elementos (.push), nunca modifica elementos existentes
- **Justificativa:** Auditoria completa exige imutabilidade de registros históricos
- **Impacto:** Funções de publicação/comprovação/avaliação sempre adicionam, nunca editam

**RN-PROP-06: Tags Categorizadas**
- **Descrição:** Proposições podem ter múltiplas tags de um conjunto pré-definido de 11 categorias
- **Justificativa:** Facilita busca e análise temática de proposições
- **Impacto:** Tags são array de IDs; renderização traduz para labels e cores

**RN-PROP-07: Prioridade com Impacto Visual**
- **Descrição:** 3 níveis de prioridade: 'urgente' (vermelho), 'alta' (laranja), 'normal' (preto)
- **Justificativa:** Destaque visual acelera identificação de proposições críticas
- **Impacto:** CSS aplica cores distintivas em badges e textos

**RN-PROP-08: Correição Obrigatória na Visualização**
- **Descrição:** Na página proposicoes.html, usuário DEVE selecionar uma correição antes de ver tabela de proposições
- **Justificativa:** Evita sobrecarga de informação; foca trabalho em uma revisão por vez
- **Impacto:** Tabela e filtros ficam ocultos até seleção de correição

**RN-PROP-09: Edição de Metadados (Não de Status)**
- **Descrição:** Modal de edição permite alterar tipo, prioridade, unidade, membro, descrição, tags - MAS NÃO status
- **Justificativa:** Status muda apenas via workflow (publicação/comprovação/avaliação)
- **Impacto:** Campo status não aparece no formulário de edição

### 4.5. Módulo de Publicação (RN-PUB)

**RN-PUB-01: Apenas Proposições Pendentes**
- **Descrição:** Página de publicação exibe SOMENTE proposições com status [' pendente', *]
- **Justificativa:** Publicação é o gateway para workflow - só aceita proposições em estado inicial
- **Impacto:** Filtro obrigatório na renderização da tabela

**RN-PUB-02: Republicação de Parcial/Em Andamento**
- **Descrição:** Proposições avaliadas como parcial ou em_andamento voltam para status 'pendente' e podem ser republicadas
- **Justificativa:** Workflow iterativo - correicionado tem nova chance de cumprir
- **Impacto:** Histórico preserva avaliação anterior; nova publicação cria novo prazo

**RN-PUB-03: Prazo Único para Lote**
- **Descrição:** Ao publicar múltiplas proposições simultaneamente, um único prazoComprovacao se aplica a todas
- **Justificativa:** Simplifica operação em lote - órgão correicionado responde tudo numa mesma data
- **Impacto:** Formulário tem um campo de prazo, não um por proposição

**RN-PUB-04: Seleção Obrigatória**
- **Descrição:** Publicação exige ao menos uma proposição selecionada (checkbox marcado)
- **Justificativa:** Operação vazia não tem sentido
- **Impacto:** Botão "Publicar" deve estar desabilitado ou validar antes de processar

**RN-PUB-05: Confirmação Antes de Publicar**
- **Descrição:** Sistema exibe dialog de confirmação mostrando quantidade de proposições e prazo antes de efetivar
- **Justificativa:** Evita publicações acidentais - operação impacta órgãos externos
- **Impacto:** Usuário pode cancelar antes de commit

**RN-PUB-06: Publicação Cria Entrada em Histórico**
- **Descrição:** Cada publicação adiciona entrada tipo 'publicacao' no historico[] com statusAnterior, statusNovo, prazoComprovacao
- **Justificativa:** Auditoria completa - saber quando, quem e com qual prazo foi publicado
- **Impacto:** Timeline visual exibe todas as publicações com ícone 📤 laranja

**RN-PUB-07: Status Muda para Aguardando Comprovação**
- **Descrição:** Após publicação, status processual muda de 'pendente' para 'aguardando_comprovacao'
- **Justificativa:** Proposição agora está com órgão correicionado esperando resposta
- **Impacto:** Proposição deixa de aparecer na lista de publicação e passa a aparecer na lista de comprovação (user)

**RN-PUB-08: Acesso Administrativo Exclusivo**
- **Descrição:** Página publicacao.html valida perfil Admin; redireciona não-admins para index.html
- **Justificativa:** Publicação é prerrogativa da Corregedoria Nacional
- **Impacto:** Usuários correicionados nunca acessam esta funcionalidade

**RN-PUB-09: Seleção de Correição Obrigatória**
- **Descrição:** Admin DEVE selecionar uma correição para visualizar proposições pendentes daquela revisão
- **Justificativa:** Organiza trabalho por correição; evita misturar proposições de revisões diferentes
- **Impacto:** Placeholder exibido até seleção; tabela só renderiza após escolha

**RN-PUB-10: Valoração Preservada na Republicação**
- **Descrição:** Ao republicar proposição com avaliação parcial/em_andamento, status se torna ['aguardando_comprovacao', valoracao_anterior]
- **Justificativa:** Valoração reflete qualidade da última avaliação - não é resetada
- **Impacto:** Histórico visual mostra que proposição já foi avaliada negativamente antes

### 4.6. Módulo de Comprovação (RN-COMP)

**RN-COMP-01: Apenas Aguardando Comprovação**
- **Descrição:** Usuário pode submeter comprovação SOMENTE para proposições com status ['aguardando_comprovacao', *]
- **Justificativa:** Workflow linear - comprovação requer publicação prévia
- **Impacto:** Dropdown de proposições filtra por este critério; proposições pendentes não aparecem

**RN-COMP-02: Submissão sem Publicação é Bloqueada**
- **Descrição:** Sistema NÃO permite submeter comprovação para proposições com status 'pendente'
- **Justificativa:** Princípio de Publication-Gated Workflow - toda comprovação precisa de prazo definido
- **Impacto:** Correicionado não consegue "adiantar" comprovação antes de receber publicação oficial

**RN-COMP-03: Múltiplos Arquivos Anexados**
- **Descrição:** Comprovação pode incluir múltiplos arquivos como evidência (faturas, fotos, atas, etc.)
- **Justificativa:** Comprovação robusta exige documentação variada
- **Impacto:** Interface de upload aceita drag-and-drop e seleção múltipla

**RN-COMP-04: Rascunhos Intermediários**
- **Descrição:** Usuário pode salvar comprovação incompleta como rascunho antes de envio definitivo
- **Justificativa:** Permite pausar trabalho e retomar depois sem perder progresso
- **Impacto:** Rascunho armazenado em proposicao.rascunhos[]; status NÃO muda

**RN-COMP-05: Um Rascunho por Proposição**
- **Descrição:** Sistema mantém no máximo 1 rascunho por proposição; salvar novo rascunho substitui anterior
- **Justificativa:** Simplifica lógica - último rascunho sempre é o válido
- **Impacto:** Array rascunhos[] tem length 0 ou 1

**RN-COMP-06: Submissão Limpa Rascunho**
- **Descrição:** Ao enviar comprovação definitiva, sistema apaga rascunho correspondente
- **Justificativa:** Rascunho perde sentido após submissão oficial
- **Impacto:** proposicao.rascunhos = []

**RN-COMP-07: Status Muda para Em Análise**
- **Descrição:** Após envio de comprovação, status processual muda de 'aguardando_comprovacao' para 'em_analise'
- **Justificativa:** Bola agora está com Corregedoria para avaliação
- **Impacto:** Proposição sai da lista de comprovação (user) e entra na fila de avaliação (admin)

**RN-COMP-08: Comprovação Cria Entrada em Histórico**
- **Descrição:** Submissão adiciona entrada tipo 'comprovacao' no historico[] com descrição, dados de anexos quando existirem, e data
- **Justificativa:** Auditoria - rastrear o que foi submetido e quando
- **Impacto:** Timeline visual exibe comprovação com ícone 📎 azul e lista de arquivos quando houver anexos

**RN-COMP-09: Submissão Após Prazo é Permitida**
- **Descrição:** Sistema aceita comprovação mesmo se prazoComprovacao < hoje
- **Justificativa:** Melhor receber atrasado do que nunca; admin pode considerar atraso na avaliação
- **Impacto:** Alerta visual de prazo vencido, mas botão de envio permanece habilitado

**RN-COMP-10: Limite de Caracteres**
- **Descrição:** Descrição de comprovação tem limite de 7.500 caracteres; observações adicionais têm limite de 1.000
- **Justificativa:** Evita textos excessivamente longos; força objetividade
- **Impacto:** Textarea com contador em tempo real; validação no submit

### 4.7. Módulo de Avaliação (RN-AVAL)

**RN-AVAL-01: Apenas Em Análise**
- **Descrição:** Fila de avaliação exibe SOMENTE proposições com status ['em_analise', *]
- **Justificativa:** Admin avalia apenas comprovações já submetidas
- **Impacto:** Tabela de avaliação filtra por este critério

**RN-AVAL-02: Quatro Decisões Possíveis**
- **Descrição:** Admin pode avaliar como: Finalizada, Parcial, Em Andamento, ou Prejudicada
- **Justificativa:** Cobre todos os cenários: sucesso total, sucesso parcial, falha, ou obsolescência
- **Impacto:** Interface exibe 4 opções visuais (cards de radio button)

**RN-AVAL-03: Parecer Obrigatório**
- **Descrição:** Avaliação exige parecer técnico escrito (até 7.500 caracteres)
- **Justificativa:** Decisão deve ser fundamentada para defesa jurídica
- **Impacto:** Campo textarea obrigatório com validação

**RN-AVAL-04: Finalizada Encerra Ciclo**
- **Descrição:** Decisão 'finalizada' muda status para ['encerrada', 'finalizada']
- **Justificativa:** Proposição cumprida - ciclo completo
- **Impacto:** Proposição sai de todas as filas de trabalho; correição pode tornar-se inativa

**RN-AVAL-05: Prejudicada Encerra Ciclo**
- **Descrição:** Decisão 'prejudicada' muda status para ['encerrada', 'prejudicada']
- **Justificativa:** Proposição obsoleta ou não mais aplicável - encerramento sem cumprimento
- **Impacto:** Similar a finalizada em termos de workflow, mas indica não-cumprimento justificado

**RN-AVAL-06: Parcial Volta para Pendente**
- **Descrição:** Decisão 'parcial' muda status para ['pendente', 'parcial']
- **Justificativa:** Cumprimento incompleto - requer nova publicação com ajustes
- **Impacto:** Proposição volta para fila de publicação; admin deve republicar

**RN-AVAL-07: Em Andamento Volta para Pendente**
- **Descrição:** Decisão 'em_andamento' muda status para ['pendente', 'em_andamento']
- **Justificativa:** Não cumprimento - requer nova publicação para dar nova chance
- **Impacto:** Proposição volta para fila de publicação; admin deve republicar

**RN-AVAL-08: Status Processual vs Valoração**
- **Descrição:** Ao avaliar como parcial/em_andamento, status processual vira 'pendente' MAS valoração preserva 'parcial'/'em_andamento'
- **Justificativa:** Status processual indica onde está no workflow; valoração indica resultado da última avaliação
- **Impacto:** Histórico preserva valoração real; badges exibem ambos

**RN-AVAL-09: Avaliação Cria Entrada em Histórico**
- **Descrição:** Avaliação adiciona entrada tipo 'avaliacao' no historico[] com parecer, decisão, statusAnterior, statusNovo
- **Justificativa:** Auditoria - rastrear decisões e fundamentações
- **Impacto:** Timeline visual exibe avaliação com ícone ⚖️ verde

**RN-AVAL-10: Republicação Preserva Histórico**
- **Descrição:** Ao republicar proposição parcial/em_andamento, histórico mantém TODAS as entradas anteriores (pub1, comp1, aval1, pub2...)
- **Justificativa:** Transparência total - mostrar quantas tentativas foram necessárias
- **Impacto:** Timeline pode ficar longa em casos de múltiplas iterações

### 4.8. Módulo de Busca (RN-BUSCA)

**RN-BUSCA-01: Busca Case-Insensitive**
- **Descrição:** Sistema ignora maiúsculas/minúsculas ao buscar texto
- **Justificativa:** Facilita uso - usuário não precisa saber capitalização exata
- **Impacto:** Implementação usa .toLowerCase() em comparações

**RN-BUSCA-02: Busca por Substring**
- **Descrição:** Sistema encontra correspondências parciais, não apenas palavras completas
- **Justificativa:** Usuário pode buscar "Proc" e encontrar "Procuradoria"
- **Impacto:** Usa .includes() em vez de ===

**RN-BUSCA-03: Busca em Tempo Real**
- **Descrição:** Resultados atualizam enquanto usuário digita (sem delay > 300ms)
- **Justificativa:** Feedback imediato melhora UX
- **Impacto:** Event listener no input com debounce opcional

**RN-BUSCA-04: Busca Combinada (AND Logic)**
- **Descrição:** Busca textual trabalha em conjunto com outros filtros (status, tags, tipo, etc.) usando lógica AND
- **Justificativa:** Refinamento progressivo - múltiplos filtros estreitam resultados
- **Impacto:** Renderização aplica todos os filtros sequencialmente

**RN-BUSCA-05: Contador de Resultados**
- **Descrição:** Sistema exibe quantidade de registros encontrados após aplicar filtros
- **Justificativa:** Feedback claro sobre efetividade da busca
- **Impacto:** Mensagem "X resultado(s) encontrado(s)" abaixo da tabela

**RN-BUSCA-06: Filtro de Status Processual**
- **Descrição:** Dropdown permite filtrar por status[0]: pendente, aguardando_comprovacao, em_analise, encerrada
- **Justificativa:** Foco em fase específica do workflow
- **Impacto:** Filtragem usa índice [0] do array bidimensional

**RN-BUSCA-07: Filtro de Valoração**
- **Descrição:** Dropdown permite filtrar por status[1]: nova, finalizada, parcial, em_andamento, prejudicada
- **Justificativa:** Foco em qualidade de cumprimento
- **Impacto:** Filtragem usa índice [1] do array bidimensional

**RN-BUSCA-08: Filtro de Tags**
- **Descrição:** Dropdown com 11 tags pré-definidas permite filtrar proposições por categoria temática
- **Justificativa:** Análise temática - ex: ver todas proposições de 'tecnologia'
- **Impacto:** Filtragem verifica se tag selecionada está em proposicao.tags[]

**RN-BUSCA-09: Limpar Filtros**
- **Descrição:** Botão "Limpar Filtros" reseta TODOS os filtros para estado padrão (vazio/todos)
- **Justificativa:** Atalho para voltar à visão completa sem limpar campo por campo
- **Impacto:** JavaScript reseta valores de dropdowns e input de busca

**RN-BUSCA-10: Campos Específicos por Contexto**
- **Descrição:** Busca textual varia conforme contexto: correições busca em numero/ramoMP/tematica/numeroElo/tipo; proposições busca em numero/tipo/unidade/membro/descricao
- **Justificativa:** Relevância - buscar nos campos que o usuário espera
- **Impacto:** Função de busca implementada separadamente em cada página

### 4.9. Módulo de Exportação (RN-EXP)

**RN-EXP-01: Dois Formatos de Exportação**
- **Descrição:** Sistema oferece exportação em JSON (dados estruturados) e PDF (documentos legíveis)
- **Justificativa:** JSON para integração/backup; PDF para impressão/compartilhamento
- **Impacto:** Menus dropdown com 2+ opções

**RN-EXP-02: JSON com Pretty-Print**
- **Descrição:** Arquivos JSON são exportados com indentação de 2 espaços (pretty-print)
- **Justificativa:** Legibilidade em editores de texto
- **Impacto:** JSON.stringify() com parâmetro indent=2

**RN-EXP-03: Metadados em Exportações**
- **Descrição:** Todos os exports incluem: titulo, dataExportacao, usuario, filtrosAplicados (quando relevante)
- **Justificativa:** Contexto - saber quando foi exportado, por quem, e com quais filtros
- **Impacto:** Objeto raiz do JSON sempre tem esses campos

**RN-EXP-04: Exportação Respeita Filtros**
- **Descrição:** Exportação captura estado atual da tabela/dashboard (com filtros aplicados), não todos os dados
- **Justificativa:** Usuário exporta o que está vendo
- **Impacto:** Funções de export usam dados filtrados, não arrays globais

**RN-EXP-05: Nomenclatura de Arquivos**
- **Descrição:** Arquivos exportados seguem padrão: `{tipo}-{data-ISO}.{ext}` (ex: dashboard-2025-12-10.json)
- **Justificativa:** Organização - evita sobrescrever exports antigos
- **Impacto:** Nome de arquivo gerado dinamicamente com new Date().toISOString()

**RN-EXP-06: PDF com Cabeçalho Institucional**
- **Descrição:** PDFs incluem header com "CONSELHO NACIONAL DO MINISTÉRIO PÚBLICO" e logo (placeholder)
- **Justificativa:** Identidade visual - documentos oficiais
- **Impacto:** HTML template para PDF inclui div.header

**RN-EXP-07: PDF Print-Optimized**
- **Descrição:** PDFs usam @media print CSS para otimizar layout na impressão
- **Justificativa:** Margens, fontes e espaçamentos adequados para papel
- **Impacto:** Styles inline no HTML do PDF

**RN-EXP-08: Ofício Formal (Times New Roman)**
- **Descrição:** Exportação de ofício de publicação usa Times New Roman 12pt com line-height 1.8
- **Justificativa:** Padrão formal de documentos jurídicos brasileiros
- **Impacto:** CSS específico para este tipo de PDF

**RN-EXP-09: Timeline Colorida em PDFs**
- **Descrição:** PDFs de relatórios completos incluem timeline com cores: laranja (publicacao), azul (comprovacao), verde (avaliacao)
- **Justificativa:** Distinção visual facilita leitura de histórico
- **Impacto:** Estilos CSS inline replicam cores da interface web

**RN-EXP-10: Validação Antes de Exportar**
- **Descrição:** Sistema valida se há dados disponíveis antes de exportar; exibe alert se vazio
- **Justificativa:** Evita exports inúteis (arquivos vazios)
- **Impacto:** if (!data.length) { alert(...); return; }

### 4.10. Módulo de Histórico (RN-HIST)

**RN-HIST-01: Estrutura de Array**
- **Descrição:** Histórico é array de objetos (historico[]) em cada proposição
- **Justificativa:** Ordem cronológica natural; facilita iteração
- **Impacto:** Renderização usa .map() ou forEach()

**RN-HIST-02: Append-Only Pattern**
- **Descrição:** Histórico nunca modifica entradas existentes - apenas adiciona novas (.push)
- **Justificativa:** Imutabilidade garante auditoria confiável
- **Impacto:** Código NUNCA deve fazer historico[i] = ..., apenas historico.push(...)

**RN-HIST-03: Três Tipos de Entrada**
- **Descrição:** Cada entrada tem campo 'tipo': 'publicacao', 'comprovacao', ou 'avaliacao'
- **Justificativa:** Diferencia natureza da interação
- **Impacto:** Renderização condicional baseada em tipo

**RN-HIST-04: Campos Condicionais**
- **Descrição:** Campos específicos por tipo: arquivos[] (comprovacao), prazoComprovacao (publicacao), parecer (avaliacao)
- **Justificativa:** Evita campos irrelevantes em certos tipos
- **Impacto:** Validação de schema deve checar presença condicional

**RN-HIST-05: Timestamps ISO**
- **Descrição:** Campo 'data' usa formato ISO 8601 (YYYY-MM-DDTHH:mm:ss.sssZ)
- **Justificativa:** Padrão internacional; preserva timezone
- **Impacto:** new Date().toISOString() ao criar entrada

**RN-HIST-06: Transições de Status**
- **Descrição:** Entradas de publicacao e avaliacao incluem statusAnterior e statusNovo (arrays bidimensionais)
- **Justificativa:** Rastrear mudanças de estado
- **Impacto:** Histórico permite reconstruir evolução completa de status

**RN-HIST-07: Ordem Cronológica**
- **Descrição:** Entradas são exibidas em ordem cronológica (mais antiga primeiro ou mais recente primeiro, conforme UX)
- **Justificativa:** Narrativa temporal lógica
- **Impacto:** Timeline renderiza array na ordem ou com .reverse()

**RN-HIST-08: Usuário Identificado**
- **Descrição:** Cada entrada registra quem fez a ação (campo 'usuario')
- **Justificativa:** Accountability - saber quem publicou/comprovou/avaliou
- **Impacto:** Valor é 'Corregedoria Nacional' (admin) ou ramoMP+ramoMPNome (user)

**RN-HIST-09: Descrição Textual**
- **Descrição:** Campo 'descricao' contém texto explicativo da ação
- **Justificativa:** Legibilidade humana - entender ação sem interpretar código
- **Impacto:** Templates de texto: "Proposição publicada para...", "Comprovação enviada por...", "Avaliação: Finalizada"

**RN-HIST-10: Observações Opcionais**
- **Descrição:** Campo 'observacoes' permite texto adicional (ou null)
- **Justificativa:** Contexto extra quando necessário
- **Impacto:** Formulários incluem textarea opcional; histórico renderiza se presente

### 4.11. Módulo de Notificações (RN-NOTIF)

**RN-NOTIF-01: Cálculo de Prazo Dinâmico**
- **Descrição:** Sistema calcula diferença entre hoje e prazoComprovacao em tempo real
- **Justificativa:** Alertas refletem urgência atual
- **Impacto:** Função executada a cada renderização

**RN-NOTIF-02: Alerta de Prazo Vencido**
- **Descrição:** Se prazoComprovacao < hoje, exibe alerta vermelho "⚠️ PRAZO VENCIDO - Vencido há X dia(s)"
- **Justificativa:** Urgência máxima - ação imediata necessária
- **Impacto:** CSS background #dc3545 (vermelho), texto branco, ícone ⚠️

**RN-NOTIF-03: Alerta de Prazo Próximo**
- **Descrição:** Se prazoComprovacao ≤ hoje+7dias, exibe alerta amarelo "⏰ PRAZO PRÓXIMO - Faltam X dia(s)"
- **Justificativa:** Aviso antecipado - tempo de agir ainda disponível
- **Impacto:** CSS background #ffc107 (amarelo), texto preto, ícone ⏰

**RN-NOTIF-04: Informação de Prazo Normal**
- **Descrição:** Se prazoComprovacao > hoje+7dias, exibe info azul "Prazo: DD/MM/YYYY - Faltam X dia(s)"
- **Justificativa:** Informação tranquila - sem urgência
- **Impacto:** CSS background #0066cc (azul), texto branco

**RN-NOTIF-05: Contador de Prazo Vencido no Dashboard**
- **Descrição:** Card "Prazo Vencido" no dashboard conta proposições com prazoComprovacao < hoje E status aguardando_comprovacao
- **Justificativa:** Visão gerencial de atrasos
- **Impacto:** Função filtra proposições por ambos critérios

**RN-NOTIF-06: Destaque Visual em Tabelas**
- **Descrição:** Linhas de proposições com prazo vencido têm background vermelho claro ou borda vermelha
- **Justificativa:** Identificação visual rápida
- **Impacto:** CSS class .prazo-vencido aplicada condicionalmente

**RN-NOTIF-07: Ícone de Rascunho**
- **Descrição:** Proposições com rascunho salvo exibem ícone "📝 Rascunho" na lista
- **Justificativa:** Indicador de trabalho em progresso
- **Impacto:** Renderização verifica proposicao.rascunhos.length > 0

**RN-NOTIF-08: Mensagens de Sucesso**
- **Descrição:** Operações críticas (publicação, comprovação, avaliação) exibem alert() de confirmação
- **Justificativa:** Feedback imediato ao usuário
- **Impacto:** alert("Proposição publicada com sucesso!") após salvar

**RN-NOTIF-09: Menção de Email (Futura Integração)**
- **Descrição:** Mensagens de sucesso mencionam "Email de notificação enviado automaticamente" (placeholder)
- **Justificativa:** Preparação para integração futura com sistema de email
- **Impacto:** Texto estático por enquanto; backend implementará envio real

**RN-NOTIF-10: Contadores em Tempo Real**
- **Descrição:** Badges e contadores (ex: "3 selecionada(s)") atualizam dinamicamente conforme ações do usuário
- **Justificativa:** Feedback imediato de estado
- **Impacto:** Event listeners chamam função updateCounter() a cada mudança
