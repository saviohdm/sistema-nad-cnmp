# Modelo de Dados
## Sistema de Acompanhamento de Proposições - CNMP

**Versão:** 1.3
**Data:** 06/04/2026
**Autor:** Documentação Técnica - NAD/CNMP

---

## 1. Introdução

Este documento descreve o modelo de dados do Sistema de Proposições do CNMP. O foco está no núcleo de acompanhamento de proposições emitidas em correições, com referência ao contexto de correições quando isso afeta a estrutura dos dados.

### 1.1 Objetivos

- Especificar entidades, estruturas internas, atributos e restrições
- Documentar relacionamentos e regras de integridade
- Servir de base para implementação de banco de dados
- Registrar decisões conceituais relevantes ao domínio

---

## 2. Visão Geral

O modelo tem **2 entidades principais** e **4 estruturas internas**:

1. **Correição**
2. **Proposição**
3. **Histórico** da Proposição
4. **RascunhoComprovacao**
5. **RascunhoAvaliacao**

### 2.1 Leitura Conceitual

- **Proposição** é o centro do workflow: publicação, comprovação, avaliação, republicação e encerramento.
- **Correição** organiza institucionalmente as proposições.
- **Histórico** é uma coleção embarcada e append-only da Proposição.
- **RascunhoComprovacao** e **RascunhoAvaliacao** são estruturas internas transitórias, com no máximo 1 item ativo de cada tipo por proposição.
- O campo `status` da Proposição representa o value object **StatusDaProposicao**.
- O campo `status` da Correição é derivado das proposições vinculadas.

## 3. Entidades

### 3.1 Correição

Representa o procedimento de correição realizado pela Corregedoria Nacional do Ministério Público.

#### 3.1.1 Atributos

| Campo | Tipo | Obrigatório | Descrição |
|-------|------|-------------|-----------|
| `id` | Integer | Sim | Identificador único da correição (PK, autoincremento) |
| `numero` | String(50) | Sim | Número da correição (ex.: `"COR-2024-01"`) |
| `ramoMP` | String(10) | Sim | Sigla do MP correicionado (ex.: `"MPBA"`) |
| `ramoMPNome` | String(200) | Sim | Nome completo do MP |
| `tematica` | Text | Não | Tema da correição |
| `numeroElo` | String(30) | Não | Identificador no sistema ELO |
| `tipo` | Enum | Sim | `'Ordinária'`, `'Extraordinária'`, `'OCD'`, `'Inspeção'` |
| `mp` | Enum | Sim | `'MPE'` ou `'MPU'` |
| `uf` | Array[String] | Sim | UFs vinculadas; 1 item para MPE, múltiplos para MPU |
| `status` | Enum | Sim | `'ativo'` ou `'inativo'`; valor derivado |
| `dataInicio` | Date | Sim | Data de início |
| `dataFim` | Date | Não | Data de encerramento |
| `observacoes` | Text(5000) | Não | Observações gerais |

#### 3.1.2 Regras de Negócio

- **RN-COR-01:** `numero` deve ser único e sequencial, reiniciando a cada ano
- **RN-COR-02:** `numeroElo` deve seguir o formato `NNNNNNN-DD.AAAA.J.TT.OOOO`
- **RN-COR-03:** Se `mp = 'MPE'`, `uf` deve conter exatamente 1 elemento
- **RN-COR-04:** Se `mp = 'MPU'`, `uf` pode conter múltiplos elementos
- **RN-COR-05:** `status` é calculado automaticamente:
  - `'ativo'`: existe ao menos uma proposição com valoração diferente de `'satisfeita'` ou `'prejudicada'`
  - `'inativo'`: todas as proposições têm valoração `'satisfeita'` ou `'prejudicada'`
  - Não pode ser alterado manualmente
- **RN-COR-06:** `dataFim >= dataInicio`
- **RN-COR-07:** UFs válidas: `AC, AL, AP, AM, BA, CE, DF, ES, GO, MA, MT, MS, MG, PA, PB, PR, PE, PI, RJ, RN, RS, RO, RR, SC, SP, SE, TO`

#### 3.1.3 Índices

- PRIMARY KEY: `id`
- UNIQUE: `numero`
- INDEX: `ramoMP`
- INDEX: `status`
- INDEX: `dataInicio`

### 3.2 Proposição

Representa uma determinação ou recomendação emitida em correição. É a unidade principal de consistência do workflow.

#### 3.2.1 Atributos

| Campo | Tipo | Obrigatório | Descrição |
|-------|------|-------------|-----------|
| `id` | Integer | Sim | Identificador único da proposição (PK, autoincremento) |
| `numero` | String(50) | Sim | Número da proposição (ex.: `"PROP-2024-0001"`) |
| `correicaoId` | Integer | Sim | FK para Correição |
| `tipo` | Enum | Sim | `'Determinação'` ou `'Recomendação'` |
| `unidade` | String(200) | Sim | Unidade responsável |
| `membro` | String(200) | Sim | Membro designado |
| `descricao` | Text(5000) | Sim | Descrição detalhada |
| `prioridade` | Enum | Sim | `'urgente'`, `'alta'`, `'normal'` |
| `prazoComprovacao` | Date | Não | Prazo definido na publicação |
| `dataPublicacao` | Date | Não | Data da publicação |
| `status` | Array[Enum] / Value Object | Sim | `[statusProcessual, valoracao]` |
| `tags` | Array[String] | Não | Tags de categorização |
| `rascunhosComprovacao` | Array[Object] | Não | Rascunhos de comprovação |
| `rascunhosAvaliacao` | Array[Object] | Não | Rascunhos de avaliação |
| `historico` | Array[Object] | Sim | Coleção append-only de interações |

#### 3.2.2 Value Object `StatusDaProposicao`

O campo `status` representa o value object `StatusDaProposicao`. Na persistência, pode ser:

- um array de 2 elementos: `[statusProcessual, valoracao]`
- ou dois campos separados: `statusProcessual` e `valoracao`

##### Status Processual

| Valor | Descrição |
|-------|-----------|
| `pendente_publicacao` | Aguarda publicação inicial ou republicação |
| `aguardando_comprovacao` | Publicada, aguarda comprovação |
| `pendente_avaliacao` | Comprovação enviada, aguarda avaliação |
| `encerrada` | Processo concluído |

##### Valoração

| Valor | Descrição |
|-------|-----------|
| `sem_avaliacao` | Valor inicial |
| `necessita_informacoes` | Comprovação insuficiente; exige novo ciclo |
| `satisfeita` | Cumprimento adequado |
| `prejudicada` | Superada ou não aplicável |

**Exemplo:**

```json
{
  "status": ["aguardando_comprovacao", "sem_avaliacao"]
}
```

#### 3.2.3 Tags Permitidas

| Tag ID | Label |
|--------|-------|
| `administrativo` | Administrativo |
| `recursos-humanos` | Recursos Humanos |
| `infraestrutura` | Infraestrutura |
| `tecnologia` | Tecnologia |
| `processual` | Processual |
| `financeiro` | Financeiro |
| `capacitacao` | Capacitação |
| `gestao-documental` | Gestão Documental |
| `compliance` | Compliance |
| `transparencia` | Transparência |
| `outros` | Outros |

#### 3.2.4 Rascunhos

`rascunhosComprovacao` e `rascunhosAvaliacao` são estruturas internas da Proposição. Não alteram o status por si só.

##### `rascunhosComprovacao`

```json
{
  "id": 1,
  "descricao": "Texto da comprovação (máx 7500 caracteres)",
  "observacoes": "Observações adicionais (máx 1000 caracteres)",
  "arquivos": ["arquivo1.pdf", "arquivo2.jpg"],
  "dataCriacao": "2024-12-01",
  "dataModificacao": "2024-12-05"
}
```

##### `rascunhosAvaliacao`

```json
{
  "id": 1,
  "decisao": "necessita_informacoes",
  "descricao": "Parecer preliminar da avaliação",
  "observacoes": "Documentação complementar ainda necessária",
  "dataCriacao": "2024-12-16",
  "dataModificacao": "2024-12-18"
}
```

**Regra:** cada proposição pode ter no máximo 1 rascunho ativo de cada tipo.

#### 3.2.5 Regras de Negócio

- **RN-PROP-01:** `numero` deve ser único
- **RN-PROP-02:** `correicaoId` deve referenciar uma correição existente
- **RN-PROP-03:** status inicial: `['pendente_publicacao', 'sem_avaliacao']`
- **RN-PROP-04:** `prazoComprovacao` e `dataPublicacao` só existem após publicação
- **RN-PROP-05:** só recebe comprovação se `statusProcessual = 'aguardando_comprovacao'`
- **RN-PROP-06:** proposições em `['pendente_publicacao', *]` não recebem comprovação
- **RN-PROP-07:** avaliação `necessita_informacoes` retorna o fluxo para `'pendente_publicacao'`
- **RN-PROP-08:** proposição encerrada tem `statusProcessual = 'encerrada'` e `valoracao ∈ {satisfeita, prejudicada}`
- **RN-PROP-09:** `tags` devem pertencer à lista predefinida
- **RN-PROP-10:** `descricao` tem limite de 5.000 caracteres
- **RN-PROP-11:** máximo de 1 rascunho ativo de comprovação por proposição
- **RN-PROP-12:** máximo de 1 rascunho ativo de avaliação por proposição
- **RN-PROP-13:** rascunhos não alteram o status; apenas submissões oficiais alteram o workflow

#### 3.2.6 Índices

- PRIMARY KEY: `id`
- UNIQUE: `numero`
- FOREIGN KEY: `correicaoId` REFERENCES `Correicao(id)` ON DELETE RESTRICT
- INDEX: `correicaoId`
- INDEX: `status[0]`
- INDEX: `status[1]`
- INDEX: `prazoComprovacao`

### 3.3 Histórico

Registra publicações, comprovações e avaliações da Proposição. É embarcado em `historico` e segue semântica append-only.

#### 3.3.1 Estrutura

`historico` é um array de objetos armazenado na Proposição.

#### 3.3.2 Atributos

| Campo | Tipo | Obrigatório | Descrição |
|-------|------|-------------|-----------|
| `tipo` | Enum | Sim | `'publicacao'`, `'comprovacao'`, `'avaliacao'` |
| `data` | DateTime | Sim | Data e hora em ISO 8601 |
| `usuario` | String(200) | Sim | Identificação do usuário |
| `descricao` | Text(5000) | Sim | Descrição da interação |
| `observacoes` | Text(1000) | Não | Observações adicionais |
| `arquivos` | Array[String] | Não | Anexos; uso típico em comprovação |
| `prazoComprovacao` | Date | Não | Prazo definido; uso em publicação |
| `statusAnterior` | Array[Enum] / Value Object | Não | Status anterior |
| `statusNovo` | Array[Enum] / Value Object | Não | Status posterior |

#### 3.3.3 Estrutura por Tipo

##### `publicacao`

Campos obrigatórios:

- `tipo`, `data`, `usuario`, `descricao`, `prazoComprovacao`, `statusAnterior`, `statusNovo`

Exemplo:

```json
{
  "tipo": "publicacao",
  "data": "2024-12-01T10:30:00.000Z",
  "usuario": "Corregedoria Nacional",
  "descricao": "Proposição publicada para MPBA - Ministério Público do Estado da Bahia",
  "observacoes": "Primeira publicação desta determinação",
  "prazoComprovacao": "2024-12-31",
  "statusAnterior": ["pendente_publicacao", "sem_avaliacao"],
  "statusNovo": ["aguardando_comprovacao", "sem_avaliacao"]
}
```

##### `comprovacao`

Campos obrigatórios:

- `tipo`, `data`, `usuario`, `descricao`

Campos opcionais:

- `arquivos`

Exemplo:

```json
{
  "tipo": "comprovacao",
  "data": "2024-12-15T14:20:00.000Z",
  "usuario": "MPBA - Ministério Público do Estado da Bahia",
  "descricao": "Implementação concluída do sistema de protocolo eletrônico em todas as promotorias da comarca.",
  "observacoes": "Incluindo capacitação de 45 servidores",
  "arquivos": ["relatorio_implementacao.pdf", "lista_presenca_treinamento.pdf", "prints_sistema.jpg"]
}
```

##### `avaliacao`

Campos obrigatórios:

- `tipo`, `data`, `usuario`, `descricao`, `statusAnterior`, `statusNovo`

Exemplo:

```json
{
  "tipo": "avaliacao",
  "data": "2024-12-20T16:45:00.000Z",
  "usuario": "Corregedoria Nacional",
  "descricao": "Comprovação analisada e aprovada. Sistema implementado conforme especificado.",
  "observacoes": "Documentação técnica completa e evidências satisfatórias",
  "statusAnterior": ["pendente_avaliacao", "sem_avaliacao"],
  "statusNovo": ["encerrada", "satisfeita"]
}
```

#### 3.3.4 Regras de Negócio

- **RN-HIST-01:** histórico é imutável; apenas inserção
- **RN-HIST-02:** entradas devem permanecer em ordem cronológica
- **RN-HIST-03:** para `publicacao`:
  - `prazoComprovacao` é obrigatório
  - `statusAnterior[0] = 'pendente_publicacao'`
  - `statusNovo[0] = 'aguardando_comprovacao'`
- **RN-HIST-04:** para `comprovacao`:
  - `arquivos` é facultativo
  - a submissão move o fluxo de `'aguardando_comprovacao'` para `'pendente_avaliacao'`
- **RN-HIST-05:** para `avaliacao`:
  - `statusAnterior[0] = 'pendente_avaliacao'`
  - se `statusNovo[1] = 'necessita_informacoes'`, então `statusNovo[0] = 'pendente_publicacao'`
  - se `statusNovo[1] ∈ {satisfeita, prejudicada}`, então `statusNovo[0] = 'encerrada'`
- **RN-HIST-06:** `usuario`:
  - publicação e avaliação: `"Corregedoria Nacional"`
  - comprovação: `"{ramoMP} - {ramoMPNome}"`
- **RN-HIST-07:** uma proposição pode ter múltiplas publicações
- **RN-HIST-08:** cada comprovação deve ser seguida de uma avaliação
- **RN-HIST-09:** `data` deve estar em ISO 8601 com timezone UTC

## 4. Relacionamentos

### 4.1 Correição → Proposição (1:N)

- Uma correição pode ter 0 ou mais proposições
- Uma proposição pertence a exatamente 1 correição
- Implementação: FK `correicaoId`
- Integridade referencial: `ON DELETE RESTRICT`
- O status da correição é recalculado a partir das proposições vinculadas

### 4.2 Proposição → Histórico (1:N Embedded)

- Uma proposição pode ter 0 ou mais entradas de histórico
- Cada entrada pertence a exatamente 1 proposição
- Implementação: array embarcado `historico`
- Regras:
  - append-only
  - toda mudança de status gera histórico
  - auditoria completa do ciclo de vida

### 4.3 Proposição → Rascunhos de Comprovação (1:N lógico, 0..1 ativo)

- Implementação: array embarcado `rascunhosComprovacao`
- Regra operacional atual: `length ∈ {0,1}`
- O rascunho é substituível
- O envio oficial consome e limpa o rascunho ativo

### 4.4 Proposição → Rascunhos de Avaliação (1:N lógico, 0..1 ativo)

- Implementação: array embarcado `rascunhosAvaliacao`
- Regra operacional atual: `length ∈ {0,1}`
- O rascunho é substituível
- A submissão oficial consome e limpa o rascunho ativo

## 6. Regras de Integridade Global

### 6.1 Integridade Referencial

- **RI-01:** toda proposição deve referenciar uma correição existente
- **RI-02:** não é permitido excluir correição com proposições vinculadas
- **RI-03:** ao excluir uma proposição, seu histórico é excluído em cascata por estar embarcado

### 6.2 Integridade de Dados

- **ID-01:** limites de caracteres devem ser validados na aplicação
- **ID-02:** datas devem ser armazenadas em ISO 8601
- **ID-03:** arrays não podem conter valores duplicados
- **ID-04:** enumerações devem ser validadas estritamente

### 6.3 Workflow

- **WF-01:** status bidimensional deve permanecer sincronizado com o histórico
- **WF-02:** mudanças de status devem gerar histórico
- **WF-03:** histórico deve ser preservado permanentemente
- **WF-04:** republicações devem criar nova publicação no histórico, sem sobrescrever a anterior
- **WF-05:** a Proposição é a unidade principal de consistência do workflow
- **WF-06:** o status da Correição deve ser recalculado após alterações relevantes em proposições vinculadas

## 7. Considerações de Implementação

### 7.1 Banco de Dados Relacional (SQL)

```sql
CREATE TABLE correicao (
  id INT PRIMARY KEY AUTO_INCREMENT,
  numero VARCHAR(50) UNIQUE NOT NULL,
  ramoMP VARCHAR(10) NOT NULL,
  ramoMPNome VARCHAR(200) NOT NULL,
  tematica TEXT,
  numeroElo VARCHAR(30),
  tipo ENUM('Ordinária', 'Extraordinária', 'OCD', 'Inspeção') NOT NULL,
  mp ENUM('MPE', 'MPU') NOT NULL,
  uf JSON NOT NULL,
  status ENUM('ativo', 'inativo') NOT NULL,
  dataInicio DATE NOT NULL,
  dataFim DATE,
  observacoes TEXT,
  INDEX idx_ramoMP (ramoMP),
  INDEX idx_status (status),
  INDEX idx_dataInicio (dataInicio)
);

CREATE TABLE proposicao (
  id INT PRIMARY KEY AUTO_INCREMENT,
  numero VARCHAR(50) UNIQUE NOT NULL,
  correicaoId INT NOT NULL,
  tipo ENUM('Determinação', 'Recomendação') NOT NULL,
  unidade VARCHAR(200) NOT NULL,
  membro VARCHAR(200) NOT NULL,
  descricao TEXT NOT NULL,
  prioridade ENUM('urgente', 'alta', 'normal') DEFAULT 'normal',
  prazoComprovacao DATE,
  dataPublicacao DATE,
  statusProcessual ENUM('pendente_publicacao', 'aguardando_comprovacao', 'pendente_avaliacao', 'encerrada') NOT NULL,
  valoracao ENUM('sem_avaliacao', 'necessita_informacoes', 'satisfeita', 'prejudicada') NOT NULL,
  tags JSON,
  rascunhosComprovacao JSON,
  rascunhosAvaliacao JSON,
  historico JSON,
  FOREIGN KEY (correicaoId) REFERENCES correicao(id) ON DELETE RESTRICT,
  INDEX idx_correicaoId (correicaoId),
  INDEX idx_statusProcessual (statusProcessual),
  INDEX idx_valoracao (valoracao),
  INDEX idx_prazoComprovacao (prazoComprovacao)
);
```

Observações:

- Arrays/objetos são armazenados como JSON
- `StatusDaProposicao` pode ser persistido em dois campos para facilitar consultas
- `status` da Correição pode ser persistido por conveniência, desde que haja recálculo confiável

## 8. Glossário

| Termo | Definição |
|-------|-----------|
| **Correição** | Procedimento de revisão judicial realizado pela Corregedoria Nacional do MP |
| **Proposição** | Determinação ou recomendação emitida durante uma correição |
| **Comprovação** | Evidência enviada pelo correicionado para demonstrar cumprimento |
| **Avaliação** | Análise da Corregedoria sobre a comprovação |
| **Rascunho** | Estado intermediário editável de comprovação ou avaliação |
| **Histórico** | Coleção cronológica e append-only das interações da proposição |
| **Status Processual** | Estado do fluxo de trabalho da proposição |
| **Valoração** | Resultado qualitativo da avaliação |
| **Status da Proposição** | Combinação entre status processual e valoração |
| **Correicionado** | Órgão do Ministério Público submetido à correição |
| **ELO** | Sistema oficial de numeração de processos administrativos do CNMP |
| **NAD** | Núcleo de Acompanhamento de Decisões |
