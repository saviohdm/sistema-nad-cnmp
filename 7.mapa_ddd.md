# Mapa DDD - Sistema NAD/CNMP

**Versão:** 2.0
**Data:** 06/04/2026

---

## 1. Domínio Central

**Acompanhar o ciclo iterativo de proposições emitidas em correições**, desde publicação até comprovação, avaliação e eventual encerramento ou republicação, com rastreabilidade completa.

**Atores principais:**
- **Corregedoria Nacional**: publica e avalia
- **Órgão Correicionado**: comprova
- **Proposição**: entidade central que evolui através do fluxo

---

## 2. Linguagem Ubíqua

**Termos Core:**
- **Correição**: procedimento de fiscalização que origina proposições
- **Proposição**: determinação ou recomendação emitida em correição
- **Publicação**: ato formal de disponibilizar proposição com prazo
- **Comprovação**: resposta do órgão com evidências de adimplemento
- **Avaliação**: julgamento da Corregedoria sobre comprovação
- **Republicação**: nova publicação após avaliação não conclusiva
- **Histórico**: trilha imutável de interações (append-only)
- **Rascunho**: estado intermediário editável (comprovação ou avaliação)

**Status Processual (fluxo de trabalho):**
- `pendente_publicacao` - aguarda publicação/republicação
- `aguardando_comprovacao` - publicada, aguarda resposta no prazo
- `pendente_avaliacao` - comprovada, aguarda decisão
- `encerrada` - concluída definitivamente

**Valoração (resultado qualitativo):**
- `sem_avaliacao` - inicial, sem avaliação
- `necessita_informacoes` - insuficiente, retorna para pendente_publicacao
- `satisfeita` - adequada, encerra ciclo
- `prejudicada` - superada, encerra ciclo

**Observação DDD:** Status bidimensional `[statusProcessual, valoracao]` deveria ser value object `StatusDaProposicao`.

## 3. Subdomínios

**Core Domain:** Ciclo de vida da proposição (publicação → comprovação → avaliação → republicação/encerramento)

**Supporting:** Gestão de Correições, Identidade/Acesso, Auditoria

**Generic:** Dashboard, busca, filtros, exportação, notificações, localStorage

## 5. Modelo Tático

### Aggregates

**Aggregate Proposição** (root principal):
- Concentra fluxo de negócio rico e invariantes críticas
- **Invariantes:**
  - pertence a exatamente uma correição
  - inicia como `pendente_publicacao + sem_avaliacao`
  - só aceita comprovação se `aguardando_comprovacao`
  - histórico é append-only
  - máximo um rascunho ativo de cada tipo
  - republicação preserva valoração anterior
- **Objetos internos:** EntradaDeHistorico, RascunhoDeComprovacao, RascunhoDeAvaliacao, StatusDaProposicao, PrazoDeComprovacao

**Aggregate Correição** (root secundário):
- Mantém consistência de atributos institucionais
- Referência para proposições
- **Nota:** campo `status` é derivado (calculado), não primário

### Entidades
- **Correição**: identidade estável, contexto institucional
- **Proposição**: identidade própria, evolui via workflow
- **EntradaDeHistorico**: entidade interna com identidade temporal
- **Rascunho**: entidade temporária substituível

### Value Objects
`StatusDaProposicao`, `NumeroElo`, `NumeroDaCorreição`, `NumeroDaProposição`, `PrazoDeComprovacao`, `Tag`, `Parecer`, `DescricaoDeComprovacao`, `ArquivoAnexado`

### Serviços de Domínio
- `PublicarProposicoesEmLote`: validar, aplicar prazo comum, alterar status, registrar historico
- `AvaliarComprovacao`: validar decisão, calcular transição, determinar encerramento/republicação
- `CalcularSituacaoDaCorreição`: inspecionar proposições, inferir ativa/inativa
- `RegistrarHistoricoDaProposicao`: garantir formato, imutabilidade, ordem cronológica

### Eventos de Domínio
`CorreicaoCadastrada`, `ProposicaoCadastrada`, `ProposicaoPublicada`, `ComprovacaoRascunhada`, `ComprovacaoEnviada`, `ComprovacaoAvaliada`, `ProposicaoRepublicada`, `SituacaoDaCorreicaoRecalculada`

## 6. Regras de Negócio Essenciais

Definição do núcleo (violá-las = alterar domínio):

1. Proposição nasce `pendente_publicacao + sem_avaliacao`
2. Publicação → `aguardando_comprovacao`
3. Comprovação → `pendente_avaliacao`
4. Avaliação `satisfeita`/`prejudicada` → encerra ciclo
5. Avaliação `necessita_informacoes` → `pendente_publicacao` (republicação)
6. Histórico imutável e acumulativo
7. Republicação preserva iterações anteriores
8. Situação da correição depende de proposições vinculadas

---

## 7. Inconsistências Identificadas

1. **Autenticação:** requisitos iniciais (produção) vs consolidado (protótipo simplificado)
2. **Avaliação flexível:** tensão entre avaliar "em qualquer status" vs fila restrita a `pendente_avaliacao`
3. **Arquivos:** são facultativos na comprovação
4. **Status bidimensional:** array funcional, mas semanticamente fraco (usar value object)
5. **Status da correição:** persistido mas derivado (risco de inconsistência)
6. **Comprovação/Avaliação:** hoje são apenas tipos de histórico, limitando evolução futura

---

## 8. Recomendações

1. Tratar `Proposição` como aggregate central
2. Nomear `StatusDaProposicao` como value object explícito
3. Manter `Histórico` no aggregate com semântica append-only
4. Tratar `status` de Correição como projeção derivada
5. Resolver contradições: avaliação fora de fluxo, autenticação protótipo vs real

---

## 9. Síntese

**Forças do modelo:**
- Fluxo de negócio claro
- Invariantes fortes
- Linguagem consistente
- Rastreabilidade via histórico

**Ajuste necessário:**
- Explicitar limites entre contextos
- Nomear conceitos como objetos de domínio
- Separar núcleo (core) de apoio (supporting/generic)

**Essência:**
- **Core:** acompanhamento iterativo de proposições
- **Aggregate principal:** Proposição
- **Contexto organizador:** Correição
- **Elemento crítico:** Histórico
- **Consumidores de estado:** Dashboard, Busca, Exportação, Notificações
